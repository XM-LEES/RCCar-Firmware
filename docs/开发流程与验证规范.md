# 下位机开发流程与验证规范

本文是 `RCCar-new/` 子仓库的强制工作流程。根目录 `../../docs/开发流程与验证规范.md` 定义跨仓库规则；本文把规则落到 STM32 固件仓库的代码路径、检查命令、测试记录和日志目录。

无论修改代码、Keil 工程、`.ioc`、文档、`.gitignore` 还是测试资料，收尾都必须执行本文的“一致性收尾检查”。

## 权威顺序

当代码、子仓库文档和根文档冲突时，按以下顺序处理：

1. 先以当前 C 源码、头文件、Keil 工程、`.ioc`、启动文件和可运行检查命令确认代码事实。
2. 再同步本仓库 `README.md`、`docs/current-state.md` 和本文。
3. 如果影响系统级阶段、架构、协议、topic、odom、测试记录或上机验收，同步根目录对应文档。
4. 不允许保留“代码一套、文档一套”的状态结束任务。

## 提交切分

推荐提交边界：

- UART4 command 解析变化单独提交。
- Telemetry 字段、状态位、帧长或 BCC 变化单独提交。
- Ackermann 执行、ESC/servo PWM 映射、限幅变化单独提交。
- RC 接管、guard、急停、刹车、通信超时变化单独提交。
- 霍尔计数、速度反馈、speed PI trim、转角估算变化单独提交。
- Keil 工程、`.ioc`、启动文件、外设初始化变化单独提交。
- 生成物清理、空目录清理、`.gitignore` 调整单独提交。
- 文档结构和流程规则修正单独提交。

跨上位机协议变化时，本仓库提交只承载下位机协议、执行、安全和本仓库文档；上位机仓库和根仓库分别提交对应桥接、消息、根文档和子模块指针。

## 本地检查命令

基础检查：

```bash
git diff --check
```

Keil 工程源码路径检查：

```bash
(
  cd MDK-ARM &&
  perl -ne 'while (m{<FilePath>([^<]+)</FilePath>}g) { print "$1\n" }' WHEELTEC.uvprojx |
  while IFS= read -r p; do
    q=${p//\\//}
    if [ ! -e "$q" ]; then echo "$p"; fi
  done
)
```

空目录检查：

```bash
find . -type d -empty -not -path './.git/*' -print
```

旧入口回归检查：

```bash
rg -n "cmd_vel|vx/vy/wz|RobotControl_task|USART3|RS485|AutoRecharge|Ranger|ICM20948|USB_HOST|BlueTooth|bsp_ServoDrive|g_orin_feedback_scale" README.md docs Core WHEELTEC_APP WHEELTEC_BSP || true
```

Keil 编译产物检查：

```bash
git status --short -- MDK-ARM/WHEELTEC WHEELTEC.bin '*.uvguix.*'
```

这些文件是本地生成物或个人 IDE 状态，不应进入功能提交。

## 当前实现与阶段目标记录要求

修改下位机协议、执行、安全或 telemetry 文档时，必须同时区分三类内容：

- 当前代码已实现：只能依据当前 C 源码、头文件、Keil 工程和可运行检查命令描述。
- 当前未实现但协议/阶段目标要求实现：必须保留在 `docs/current-state.md` 和根 `../../docs/接口与协议.md` 中，标为待补齐或未验收。
- 验收前必须补齐：必须写清需要补的代码路径、状态位、测试场景和测试记录要求。

阶段 1 当前已实现的下位机事实包括 Ackermann command 解析、速度/转角执行端限幅、RC 接管、急停、刹车、通信超时、霍尔反馈、24 字节 telemetry，以及已填充的安全/反馈状态位。

以下内容不得在文档中删除或改写成“不需要做”：

- 完整 `FAULT_LATCHED` 和 `CLEAR_FAULT` 锁存故障清除语义。
- 完整 `status_flags`/`status_bits` 目标表。
- `SPEED_SATURATED`、`STEERING_SATURATED`、`ACCEL_LIMITED`、`STEERING_RATE_LIMITED` 等限幅状态位。
- 加速度限幅和转角速度限幅。
- 真实转角测量能力对应的 `STEERING_IS_MEASURED`，除非当前硬件事实确认不存在并在根阶段计划中调整目标。

未实现位在补齐前必须保持为 `0`，不得在测试记录、提交说明或 README 中宣称已经具备。若阶段验收时仍未补齐，验收记录必须把它列为未通过项或明确的后续阻塞项，而不是从目标中移除。

## 一致性收尾检查

每次修改结束前都必须执行：

```bash
git diff --check

(
  cd MDK-ARM &&
  perl -ne 'while (m{<FilePath>([^<]+)</FilePath>}g) { print "$1\n" }' WHEELTEC.uvprojx |
  while IFS= read -r p; do
    q=${p//\\//}
    if [ ! -e "$q" ]; then echo "$p"; fi
  done
)

find . -type d -empty -not -path './.git/*' -print

rg -n "<本次改动关键词>" \
  README.md docs Core WHEELTEC_APP WHEELTEC_BSP MDK-ARM WHEELTEC.ioc \
  ../README.md ../AGENTS.md ../docs || true
```

检查结论必须写进最终回复或提交说明：

- 代码事实是否已经反映到 `README.md` 和 `docs/current-state.md`。
- 根 `docs/接口与协议.md` 是否需要同步。
- 根 `docs/系统架构与数据流.md` 是否需要同步。
- 根 `docs/开发流程与验证规范.md` 是否需要同步。
- 若无需同步，明确写“已检查上层文档，无需更新”。
- 若检查发现冲突，以代码事实为准修正文档后再结束。

## 测试记录和日志目录

整理后的测试记录放：

```text
docs/test-records/
```

原始日志和临时调试输出放：

```text
logs/
```

规则：

- `docs/test-records/` 存放可读、可提交的 Markdown 记录，例如静态检查结果、Keil 编译烧录摘要、架空测试和低速实车测试摘要。
- `logs/` 存放原始串口抓包、Keil build log、终端输出、临时调试文件和大体积附件；默认不提交。
- 需要长期保留的原始日志，先压缩并在 `docs/test-records/*.md` 中写明路径、采集命令和摘要，再决定是否单独提交。
- 实车运动相关记录必须写明测试场景、是否架空、命令输入、实际输出、是否触发安全状态。

推荐记录命名：

```text
docs/test-records/YYYY-MM-DD-<scope>-<result>.md
logs/YYYY-MM-DD-<scope>-<source>.log
logs/YYYY-MM-DD-<scope>-serial.bin
```

## 编译验证要求

以下改动影响固件产物，提交前必须给出编译结论：

- C 源码、头文件、启动文件、中断、任务、外设初始化。
- Keil 工程、CubeMX `.ioc`、RTE、编译选项、源码路径。
- 会改变协议、执行、安全、telemetry 或反馈语义的配置宏。

在 Windows/Keil 环境完成上述改动时，用于功能合入的提交必须通过 Keil
Build。编译失败的提交只能用于保存失败调查状态，不能标记为可烧录或可合入。

在无 Keil 工具链环境完成上述改动时，不强制编译；必须执行本地静态检查，
并在最终回复、提交说明或测试记录中写明未编译原因，例如
`未编译：当前环境无 Keil 工具链`。

纯文档、日志整理、流程规范、调参 skill、非固件配置和测试模板改动不要求
编译，仍必须执行一致性收尾检查。

## Windows/Keil 编译烧录

Ubuntu 开发环境不执行下位机固件编译和烧录。用户切换 Windows 后，使用 Keil 工程：

```text
MDK-ARM/WHEELTEC.uvprojx
```

Windows/Keil 环境下，影响固件产物的功能提交至少执行 Build；烧录和上电
验证按测试目标执行。

Windows/Keil 编译烧录记录至少包含：

- 日期、操作者、当前分支和提交。
- Keil 编译结果。
- 固件产物路径。
- 烧录结果。
- 上电后的关键现象。
- 是否架空、是否触发 RC 接管/急停/刹车/通信超时。
- 记录对应的 `logs/` 原始文件路径。
